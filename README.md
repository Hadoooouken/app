# 2D планировщик квартиры (JavaScript + SVG)

D-планировщик квартиры на чистом JavaScript.  
Рисование и отрисовка выполнены через SVG. Проект позволяет собирать планировку из капитальных и обычных стен, вставлять двери и окна, размещать мебель, считать комнаты и метрики, а также работать с историей изменений.

---

## Что умеет проект

- строить план квартиры в 2D;
- работать с **капитальными** и **обычными** стенами;
- вставлять окна и двери в стены;
- размещать мебель из SVG-спрайтов;
- двигать, вращать и выделять мебель;
- автоматически определять комнаты;
- считать размеры и площади;
- поддерживать `undo/redo`;
- работать с pan/zoom;
- иметь отдельную логику под desktop и mobile.

---

## Технологии

- **JavaScript (ES modules)**
- **SVG**
- **svg.js**
- без тяжелого фреймворка

## Базовые единицы измерения

В проекте используются **world units**, где:

- `1 unit = 1 см`
- `100 units = 1 метр`

Примеры:

- `200` = `2.00 м`
- `90` = `0.90 м`
- `250` = `2.50 м`

Это важно для всех размеров стен, мебели, дверей, окон и расчетов комнат.

# Как устроены стены

В проекте есть **два типа стен**:

## 1. Капитальные стены (`capital`)

Это внешний или базовый контур помещения.

Они задают:

- основную форму квартиры;
- границы допустимой зоны;
- опорную геометрию для внутренних стен;
- основу для расчета комнат.

### Геометрия капитальных стен

У капитальной стены используется **видимая геометрия** и **внутренние грани**.

Условно:

- `a/b` — край стены;
- `ia/ib` — внутренняя сторона стены.

Внутренние грани достраиваются через `ensureCapitalInnerFaces()`.

Это нужно для того, чтобы:

- понимать внутренний контур помещения;
- корректно вставлять объекты;
- запрещать выход обычных стен за допустимую область;
- строить комнаты по внутреннему контуру.

---

## 2. Обычные стены (`normal`)

Это пользовательские перегородки, которые рисуются внутри капитального контура.

### Геометрия обычных стен

У обычной стены в проекте фактически две геометрии:

- `va/vb` — **строительная ось**
- `a/b` — **видимая подрезанная геометрия**

Нормализация выполняется через `normalizeNormalWall()`.

Это сделано специально.

### Зачем нужны две геометрии у обычной стены

Потому что редактору нужно одновременно решать две разные задачи:

#### Строительная логика

Для снапа, соединений, Т-стыков и ограничений удобнее работать по оси стены.

#### Визуальная логика

На экране стену нужно красиво подрезать по стыкам, чтобы:

- не было лишних хвостов;
- не было наложений;
- Т-стыки выглядели корректно;
- комнаты считались по реальному видимому контуру.

Именно поэтому ось и видимая часть разделены.

---

## Почему это важно

Без разделения геометрии редактор быстро начинает ломаться:

- снап работает в одной точке, а рендер рисует в другой;
- стыки выглядят криво;
- комната считается не по тому контуру;
- двери/окна визуально “едут”.

Текущая схема решает эту проблему правильно.

# Ограничения и снап

За правила построения отвечает `constraints.js`.

В проекте есть логика:

- умного снапа (`smartSnapPoint`);
- Т-стыка;
- запрета пересечений;
- проверки нахождения внутри допустимой зоны;
- проверки отступа от капитальных стен;
- проверки чистоты от конфликтов с существующей геометрией.

Типовые ограничения:

- обычная стена должна быть внутри допустимого контура;
- нельзя строить некорректные пересечения;
- нельзя влезать в запретные зоны;
- нельзя ставить некоторые объекты слишком близко к геометрии, если это запрещено правилами.

---

# Как считаются комнаты

За комнаты отвечает `computeRooms()`.

Общий принцип такой:

1. Из стен собираются грани и контуры.
2. Формируется замкнутая геометрия помещения.
3. Определяются внутренние полигоны комнат.
4. Для каждой комнаты считается площадь.
5. Для подписи комнаты берется удобная внутренняя точка (через `polylabel`).

### Что это дает

Проект умеет:

- находить комнаты автоматически;
- показывать площадь;
- ставить подпись не в стену и не за пределы полигона, а в “хорошую” внутреннюю точку.

---

# Как рендерится сцена

Основной рендер делает `render()`.

Обычно рендер состоит из таких логических частей:

## 1. Фон и сетка

Рисуется фон, сетка и pattern-grid.

## 2. Основная геометрия

Рисуются:

- капитальные стены;
- обычные стены;
- двери;
- окна;
- мебель.

## 3. Оверлеи

Рисуются:

- выделение;
- hover;
- активные хэндлы;
- превью перед установкой;
- размеры;
- подписи комнат;
- курсорные подсказки.

## 4. Viewport-преобразование

К сцене применяются:

- `scale`
- `offsetX`
- `offsetY`

То есть весь мир рисуется в world units, а потом показывается через текущий zoom/pan.

---

## Ключевая идея рендера

Рендер **не должен хранить бизнес-логику**.  
Он берет текущее состояние и показывает его.

То есть:

- состояние и геометрия считаются отдельно;
- `render()` отображает уже готовый результат.

Это делает проект проще в поддержке.

---

# Двери и окна

Двери и окна вставляются в стены как SVG-символы.

### Важно

Они не просто “лежат сверху”, а привязываются к стене:

- знают `wallId`;
- знают позицию по стене;
- проходят проверки допустимости;
- рендерятся в ориентации стены.

Для превью используется временное состояние, например `previewDoor`.

---

# Мебель

Мебель работает через SVG-спрайты.

У каждого предмета обычно есть:

- `typeId`
- `label`
- `symbolId`
- `w`
- `h`
- позиция
- угол поворота

Каталог мебели задается конфигом наподобие `FURN_CATEGORIES`.

Пример структуры:

- категория;
- список предметов;
- у каждого предмета размеры в сантиметрах;
- SVG-символ, которым он рисуется.

---

## Как рисуется мебель

При постановке мебели:

1. выбирается элемент из каталога;
2. создается объект в `state.furniture`;
3. ему задается позиция, размер, угол;
4. рендер берет `symbolId` и рисует нужный `<use>` из SVG-спрайта;
5. при выборе предмет получает рамку/хэндлы/поворот.

---

# История изменений

Проект поддерживает `undo/redo`.

Обычно для этого используются снапшоты состояния:

- `historyBegin()`
- `historyCommit()`
- `historyEnd()`
- `undo()`
- `redo()`

Идея простая:

- перед изменением начинается операция;
- после изменения состояние фиксируется;
- пользователь может откатить или вернуть шаг.

Это особенно важно для редактора, где много мелких действий.

# Режимы работы

В `state.mode` обычно лежит активный режим.

Например:

- `select`
- `draw-wall`
- `draw-door`
- `draw-window`
- `place-furniture`

От режима зависит:

- что делает pointerdown;
- есть ли превью;
- можно ли рисовать новую стену;
- выделяется ли объект;
- ставится ли мебель по тапу.

# Почему проект устроен именно так

Это не просто “рисовалка”, а маленький геометрический редактор.

Поэтому здесь важно разделение:

- **данные**
- **геометрия**
- **ограничения**
- **отрисовка**
- **взаимодействие**

Если смешать это в одном месте, проект быстро становится неуправляемым.

Текущая архитектура хороша тем, что:

- ее можно расширять;
- проще искать баги;
- можно добавлять новые типы мебели и объектов;
- можно отдельно улучшать рендер и отдельно математику.
